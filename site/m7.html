A feel for each kernel

What are these things, actually?

The names are mangled C++ templates, but each one maps to something concrete in the transformer. They sort into three families: tensor-core heavyweights doing the real math, memory-bound streamers that just move bytes (the fusion headroom), and fused specialists that already do their job in one shot.

Tensor-core heavyweights — the real FLOPs (~42%)

  • cutlass … s16816gemm 24.9% — the big FFN matrix multiply. Every layer's feed-forward block expands the 1280-dim hidden state to ~5120 and back; this is NVIDIA's hand-tuned CUTLASS GEMM doing those bf16 matmuls on the tensor cores. s16816 is the tensor-core instruction shape (16×8×16). It's the single most expensive kernel — and it's the good kind of expensive: dense work running near peak. The relu in the name is a fused epilogue, so the activation rides along for free.
  • ampere … s1688gemm 12.6% — the other GEMM family: the attention Q/K/V and output projections and the smaller linears. Same tensor cores, a different tile shape (s1688 = 16×8×8) that the autotuner picked for these dimensions.
  • flash_fwd_kernel 4.7% — flash-attention. The whole softmax(QKᵀ)V is fused into one kernel that tiles over the sequence and keeps the attention matrix on-chip — it never writes the full N×N scores to HBM. That fusion is exactly why "the attention" is so cheap here despite the quadratic reputation.

Memory-bound streamers — the fusion headroom (~43%)

These do almost no arithmetic. Each reads a tensor from HBM, touches every element once, and writes it back — so they're limited by bandwidth, not compute. Launching them separately means the same activations get streamed through memory over and over. This is precisely what torch.compile fuses into the neighbouring kernels.

  • elementwise · mul 8.1% — per-element multiply: scalings, gates, the multiplicative half of GeLU.
  • elementwise · add 5.9% — per-element add: the x + sublayer(x) residual connections and bias adds.
  • elementwise · erf 5.3% — the GeLU activation, which is built on the error function erf. Transcendental math per element, but at this size still dominated by the read/write.
  • direct_copy (cast) 5.6% — a dtype-conversion copy (bf16↔fp32). Some ops upcast to fp32 for numerical safety and come back; this kernel exists only to stream the tensor through memory to change its type. Pure overhead — a prime fusion target.
  • CatArrayBatchedCopy (rotary) 4.4% — a concatenate from the rotary position embeddings (RoPE rotates query/key halves and stitches them back with a cat). No math, just data movement.

Fused specialist — already a single kernel (~1%)

  • vectorized_layer_norm 1.3% — LayerNorm: compute mean/variance across the hidden dim and normalize. It's a reduction plus an elementwise pass, but PyTorch ships it as one fused kernel, so it's already cheap and there's little left to win.
The one-line intuition

The GEMMs and flash-attention are doing the actual work and running efficiently. The other ~43% is the model shuffling activations through HBM between those matmuls — and that shuffling, not the math, is the headroom a fused/compiled backend reclaims.

What the profile says

Key takeaways

  • Attribution checks out. Leaf-kernel self-times sum to 225 ms/iter, equal to the Milestone 2 median — the forward runs essentially serially on one stream.
  • Attention is not the bottleneck here. The fused flash-attention kernel is only ~4.7%; the matmul cost is the FFN/projection linears (24.9% + 12.6%). Optimizing attention further would barely move this shape.
  • ~43% is unfused elementwise — the fusion headroom. GeLU, bias-adds, residual adds, and scalings each launch their own kernel and stream activations through HBM. That is exactly what torch.compile fuses, and it explains the 1.63× compile speedup at this same shape in Milestone 5.
  • ~14% is copies/casts (rotary cat, dtype casts); LayerNorm ~1%. Memory is ~1.4 GB of 24 GB — not capacity-bound.
Honest scope

Limitations

Read the buckets as order-of-magnitude

The category rollup is heuristic (kernel-name substrings): the matmul-vs-rest split is robust, but "elementwise" bundles activation/bias/residual/scale and "copy" bundles casts and concatenations. It is eager-only, one shape (seq 512 × batch 8), on this L4 + PyTorch 2.12.0+cu130 stack — a compiled run, a different shape, or another GPU gives a materially different split.
