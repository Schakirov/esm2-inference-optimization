Making ESM2 protein embeddings fast & correct on an NVIDIA L4

A reproducible study of inference performance for the protein-language model facebook/esm2_t33_650M_UR50D (650M params) on a single power-efficient L4 GPU. The work proceeds through ten human-reviewed milestones (0–9) — honest baselines, correctness checks, batching/padding analysis, torch.compile, a custom Triton pooling kernel, and profiler evidence — prioritizing reproducibility over headline numbers.

The task setting

Why this project exists

Protein-language models like ESM2 are a standard front-end for biotech ML: you embed a protein sequence once and reuse the vector for downstream tasks (function prediction, variant effect, structure, retrieval). The embedding step is pure inference and is run over millions of sequences, so its throughput and cost matter directly.

This project documents the unglamorous-but-real engineering of making that step fast and correct on commodity inference hardware. The L4 is a common, power-efficient inference GPU (72 W class), which makes ESM2 embedding throughput on it a realistic target.

Why ESM2 & protein embeddings specifically

  • Open & standard — widely used, available through Hugging Face transformers.
  • Representative workload — variable-length sequences, padding, attention, per-token hidden states, and a pooling step to produce one vector per protein.
  • Exposes classic perf themes — dtype choice, batching, padding waste, static vs dynamic shapes, kernel fusion.
What this project does not claim

It does not claim to beat vendor-optimized stacks (TensorRT, FlashAttention kernels) unless explicitly benchmarked; it does not claim the custom Triton kernel is universally faster than PyTorch; and all numbers are specific to this L4 + software stack and a synthetic sequence distribution. Negative results are reported honestly.

Why ~22 GiB vs 23034 MiB?

How numbers are produced

Methodology & design principles

  • CUDA events, not wall clock. Each timed iteration is bracketed by a torch.cuda.Event pair; we synchronize() then read elapsed_time. Kernel launches are async, so time.time() would measure launch overhead, not device execution.
  • Warmup + steady state. Untimed warmup iterations absorb lazy CUDA init and cuDNN/cuBLAS autotuning; median latency is the headline (robust to outliers), with mean/std also recorded.
  • Honest memory. reset_peak_memory_stats() before timing, max_memory_allocated() after; models are explicitly freed between dtypes so peaks aren't contaminated.
  • Real vs padded tokens. Throughput is reported as real amino-acid tokens/sec, not just padded compute tokens — padding waste is a first-class metric.
  • Correctness first. Optimized paths are validated against an fp32 eager reference with documented tolerances before any speed claim.
  • Graceful failure. OOM is caught per cell and recorded (oom=True), never masked as success.
  • Repo hygiene. No weights, venvs, caches, secrets, or large traces committed — only small summarized artifacts. Commit locally, never push.
