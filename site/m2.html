What the baseline says

Key takeaways

  • Overhead floor. At small total work (seq 128, batch 1–2) latency is pinned near ~20 ms regardless of token count — the regime is launch/overhead-bound, not compute-bound.
  • Throughput saturates. Real-token throughput climbs with batch/length and plateaus around 21–22k tok/s, then dips slightly at the largest cells as attention's O(L²) cost grows.
  • bf16 ≈ fp16. Within noise on Ada Tensor Cores; bf16 marginally faster at the largest shapes. Memory is identical (both 2-byte weights). Numerical divergence — not throughput — is the story, and that's Milestone 3.
  • Far from memory-bound. Peak ~2.0 GB of 24 GB; no OOM in the whole matrix. Larger batches/lengths are headroom for later milestones.
  • Upper-bound caveat. Numbers are encoder-forward-only — no tokenization, pooling, or I/O — so a real embedding pipeline will be slower.
An honest war story

A real measurement bug, found & fixed

Cross-dtype peak-memory contamination

The first run reported fp16 peak memory (~2.4 GB) as roughly double bf16 (~1.2 GB), despite both storing 2-byte weights. Root cause: the bf16 model wasn't released before the fp16 model loaded, so the second dtype's peak counted both resident weight sets. Fix: explicitly free the previous model (del; gc.collect(); empty_cache(); synchronize()) between dtypes. After the fix both dtypes report identical ~1.2 GB at equal shapes — exactly the kind of artifact that silently inflates a memory comparison.

Encoder vs LM head

Loaded via AutoModel (the EsmModel encoder); we read only last_hidden_state. The lm_head.* reported UNEXPECTED and the randomly initialized pooler.* reported MISSING at load are both harmless and intentionally unused.
