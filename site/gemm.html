Inside one s16816gemm: a tile-by-tile walk on the L4

The top kernel in the Milestone 7 profile is cutlass_80_tensorop_bf16_s16816gemm_relu_bf16_256x128_32x3_tn_align8 — the ESM2 FFN matrix multiply, 24.9% of the forward. This page follows one real instance of that kernel doing the 650M model's feed-forward up-projection, and answers, at every level of the hardware: which block, which warp, which thread touches which slice of A, B and C — and where each number physically lives (HBM, L2, shared memory, or registers) when the tensor core consumes it. No simulation — just the logic of the tiling, with real L4 numbers.

The concrete case

One matrix multiply, real shapes

ESM2-650M has embedding width d=1280 and FFN width 4d=5120. At the profiled shape (seq 512 × batch 8 = 4096 tokens), the FFN up-projection is a single GEMM C = A · B:

  • A = activations X, M×K = 4096×1280 bf16 — one row per token.
  • B = weight W₁, K×N = 1280×5120 bf16 — the up-projection matrix.
  • C = output Y, M×N = 4096×5120 — accumulated in fp32, written as bf16.

That is 2·M·N·K = 53.7 GFLOP for this one matmul. At the L4's 121 TFLOP/s bf16 peak the arithmetic floor is ~0.44 ms; the minimum HBM traffic if every element of A, B, C is touched once is only 65.5 MB (~0.22 ms at 300 GB/s). The ratio — ~819 FLOP per byte — is what makes this compute-bound, and is the entire reason for the tile hierarchy below: it exists to turn that 819 into reality by keeping operands in fast memory and feeding the tensor cores, instead of re-reading A and B from HBM thousands of times.

The shape of the work

What the name tells us vs. what is the standard config

The CTA tile (256×128×32), stage count (3), and instruction shape (16×8×16) are read directly from the kernel name. The warp shape (64×64) and thread count (256) are not encoded in the name — they are the canonical CUTLASS pairing for a 256×128×32 bf16 tile (4×2 warps), which is what cuBLAS uses for this kernel. Treat the warp-level numbers as the standard configuration, not as something the profiler proved for this exact run.

Where things live

The L4 memory hierarchy this kernel rides

Every value A, B and C starts and ends in HBM. The tile hierarchy is exactly a plan for staging those values up the pyramid into faster, smaller storage so the tensor cores never wait on HBM:

L4: 58 SMs, 4th-gen tensor cores (4/SM), 128 KB combined L1/shared per SM (up to 100 KB as shared), 64K 32-bit registers/SM (max 255/thread), 48 MB L2, 24 GB GDDR6 @ 300 GB/s, CC 8.9.

Level 1 · the grid

640 blocks, ~11 waves across 58 SMs

C is 4096×5120. Cut into 256×128 patches that is (4096/256) × (5120/128) = 16 × 40 = 640 blocks. Block (bm, bn) is solely responsible for C[bm·256 : +256, bn·128 : +128] — e.g. block (0,0) owns C's top-left 256×128 corner; block (15,39) owns the bottom-right.

The L4 has 58 SMs. This kernel's block needs ~72 KB shared and ~255 registers/thread, so only one block fits per SM at a time. 640 blocks ÷ 58 SMs ≈ 11 waves: the GPU runs 58 blocks at once, they finish, the scheduler launches the next 58, eleven times over. Operand reuse is across blocks, and that is what L2 is for: the 40 blocks sharing a row-band of A all read the same 256×1280 A-panel (640 KB); the 16 blocks sharing a column-band of B all read the same 1280×128 B-panel (320 KB). Both panels are tiny against the 48 MB L2, so after the first block pays the HBM cost the rest are served from L2 — the effective operand bandwidth is far above the raw 300 GB/s.

Level 2 · one block on one SM

The mainloop: stage, compute, repeat — 40 times

Take block (0,0). It must compute C[0:256, 0:128] = A[0:256, 0:1280] · B[0:1280, 0:128]. K=1280 is too big to hold, so the block walks K in 40 steps of 32. Each step it stages a thin slab of A and B into shared memory and feeds them to the tensor cores:

HBM → sharedA-slab A[0:256, k:k+32] (16 KB) and B-slab B[k:k+32, 0:128] (8 KB) are copied straight from L2/HBM into shared by cp.async, bypassing the register file. 256 threads cooperate on the coalesced load (128-bit each, align8).
3-stage pipelineWhile the tensor cores chew on stage k, cp.async is already pulling stages k+1 and k+2. Three slabs of each operand live in shared at once → 3 × (16+8) = 72 KB, inside the L4's 100 KB budget. This is how the 300 GB/s latency is hidden behind compute.
shared → registersEach warp issues ldmatrix to pull its operand fragments from shared into registers, already permuted into the layout the tensor cores require.
registers → tensor coreThe warp issues its mma.sync instructions; results accumulate in place in each thread's fp32 registers. Nothing returns to shared or HBM until the very end.

After 40 iterations the contraction over K is complete and every thread holds its final fp32 partial sums in registers. The 256×128 = 32,768 outputs of this block live entirely in the registers of its 256 threads (128 each).

To build its 64×64, a warp needs the matching 64 rows of the A-slab and 64 cols of the B-slab sitting in shared. Within one K-step of 32, it covers K in two MMA sub-steps of 16, and for each it issues (64/16) × (64/8) = 4 × 8 = 32 mma.sync instructions — so 64 MMAs per warp per mainloop iteration, ×40 iterations = 2,560 MMAs to finish its tile. All 32 threads of the warp execute each mma.sync together; the instruction is warp-wide.

Level 4 · one instruction

One mma.sync.m16n8k16, thread by thread

This is the atom. mma.sync.aligned.m16n8k16.row.col.f32.bf16.bf16.f32 takes a 16×16 tile of A and a 16×8 tile of B and produces a 16×8 tile of C, contracting the 16-long K. The 32 threads of the warp split the operands and the result among themselves — there is no shared scratch inside the instruction, everything is in registers:

  • A fragment (16×16 bf16 = 256 values): each thread holds 8 bf16, packed in 4 32-bit registers.
  • B fragment (16×8 bf16 = 128 values): each thread holds 4 bf16, in 2 registers.
  • C/D fragment (16×8 fp32 = 128 values): each thread holds 4 fp32 accumulators — the ones it has been carrying across all 40 K-steps.

The result ownership is fixed by the PTX layout: thread t sits in group g = t÷4 and lane ℓ = t mod 4, and owns the four C elements at rows {g, g+8} × cols {2ℓ, 2ℓ+1}. The full 16×8 map — every cell labelled with the thread that holds it (note each thread appears in two rows, r and r+8):

So thread 0, across this one instruction, computes and accumulates exactly four outputs — C[0,0], C[0,1], C[8,0], C[8,1] of the micro-tile — by taking the dot product of its share of the A-rows with its share of the B-cols, all from registers, in a single hardware op. The tensor core reads A and B fragments from registers and adds into the C registers; L2 and HBM are nowhere on this path. Multiply that by 32 micro-tiles per warp × 8 warps × 2 K-substeps × 40 iterations and the block's whole 256×128 patch is built.

The finish

Epilogue: registers → shared → HBM

When K is exhausted, each thread's 128 fp32 accumulators are the final answer for its slice of C. The epilogue then: applies the alpha scale (and the linear's bias if fused), casts fp32 → bf16, and writes back to C in HBM. Because the per-thread register layout is scattered (those r, r+8 stripes), the values are first shuffled through shared memory into contiguous rows so the store to HBM is fully coalesced. The relu in the kernel name could run here for free — but ESM2's GeLU does not, so it lands a few kernels later as the separate erf op you see in the Milestone 7 breakdown.

The one-sentence version: A and B climb HBM → L2 → shared → registers once per tile and get reused hundreds of times up there; C is born in registers, accumulated entirely in registers across 40 K-steps, and only touches HBM once, at the end. That staged reuse is what converts an 819-FLOP/byte compute-bound problem into a kernel that actually runs near the L4's tensor-core peak.

Honest scope

What's certain, what's the standard config

  • Read from the kernel name (certain): bf16 in/out, m16n8k16 instruction, 256×128 CTA tile, K-step 32, 3 pipeline stages, tn layout, 128-bit aligned loads.
  • The canonical CUTLASS config for that tile (not in the name): 64×64 warp shape → 8 warps / 256 threads, and the resulting 72 KB shared / ~255 reg-per-thread / 1-block-per-SM occupancy. These are the standard pairing cuBLAS uses, but the profiler did not dump the internal warp shape for this run.
  • The MMA fragment & C-ownership layout is exact — it is fixed by the PTX ISA for mma.m16n8k16.f32, identical on every Ampere/Ada/Hopper tensor core.
  • L4 numbers (58 SM, 100 KB shared, 64K regs, 48 MB L2, 300 GB/s) are the published Ada / L4 specs; the per-SM occupancy limits come from compute capability 8.9.
  • Real cuBLAS may also pick split-K or stream-K variants for some shapes; this walk assumes the straightforward data-parallel tiling the named kernel implements.
