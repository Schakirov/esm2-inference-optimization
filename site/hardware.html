The L4 in context: GPUs, and how ESM2 was trained

This project optimizes inference for facebook/esm2_t33_650M_UR50D on a single NVIDIA L4. This page situates that choice: how the L4 compares to the datacenter training/inference GPUs (A100, H100, and the Blackwell B100/B200/B300), how the ESM2 family was actually trained and at what cost, a back-of-envelope of how long that training would take on each chip, and why an L4 is the right target for an inference-optimization portfolio.

Correction — SM count is not a mystery

An earlier version of this page implied NVIDIA "doesn't publish" SM counts, so they're unknowable. That's wrong: the SM count is a runtime-queryable property of any CUDA device you hold — cudaDeviceGetAttribute(&n, cudaDevAttrMultiProcessorCount, dev), or just deviceQuery / nvidia-smi -q, reports the actual enabled count. The L4 figures above were read straight off this machine. The Blackwell ~ values are third-party estimates only because those cards aren't in this machine to query and NVIDIA's datasheets don't list per-die SM counts — anyone holding a B200 can read its real count the same one-line way.

The model

How ESM2 was trained

ESM2 is a family of BERT-style protein language models from Meta AI (FAIR), trained with a masked-language-modeling objective over evolutionary-scale protein sequence data. The headline result — atomic-resolution structure prediction (ESMFold) — came from scaling this masked LM to 15B parameters. The model this project optimizes, esm2_t33_650M, is the 33-layer / 650M-parameter member.

  • Data: UniRef50 (release 2021_04). Each training example is a UniRef50 cluster representative, but the actual sequence is resampled from that cluster's UniRef90 members each time it is seen — so the model sees roughly 65 M unique sequences drawn from ~43 M UniRef50 clusters (~138 M UniRef90 sequences). Sequences are cropped to a 1,024-token context.
  • Objective: masked language modeling — 15% of residues are masked and predicted from context. No structure or MSA supervision; structure emerges from sequence statistics alone.

Source & honesty note

Paper: Lin et al., "Evolutionary-scale prediction of atomic-level protein structure with a language model," Science 379:1123–1130 (2023) (bioRxiv preprint). The GPU/time/FLOP figures come from an ESM maintainer's estimate in facebookresearch/esm discussion #414, not the paper text, and are explicitly back-of-envelope (≈45 TFLOP/s sustained per V100). The 15B model's GPU count and wall-clock time are not publicly stated — only the ~1.2e23 FLOP estimate — so they are left blank rather than guessed. Dataset/objective details are corroborated via the Hugging Face model cards and secondary quotations of the paper's methods.

Read these as order-of-magnitude only

This counts compute alone. It ignores memory capacity, memory bandwidth, interconnect, and parallel-scaling efficiency — all of which matter enormously at 512-GPU scale. The L4 figure is especially academic: at 24 GB and 300 GB/s the L4 is an inference card, and a real 650M training run would be bottlenecked by memory and communication long before compute, so "~same as a V100" understates how unsuitable it actually is. Newer chips would also use FP8 (Hopper/Blackwell) or FP4 (Blackwell) to roughly halve or quarter these numbers again — but ESM2 trained in fp16, so BF16 is the fair comparison.

The point

Why an L4 for this project?

  • The task is inference, not training. Extracting embeddings from a pretrained 650M model is a forward-only, latency- and throughput-sensitive serving workload — exactly what the L4 is built for. You would never train ESM2 on an L4, and you don't need an H100 to serve it.
  • It fits, with room to batch. 650M weights in bf16 are ~1.3 GB; the Milestone 7 profile used ~1.4 GB of the L4's 24 GB at seq 512 × batch 8. The capacity that makes the L4 a poor training card is more than enough for inference.
  • Cheap, low-power, ubiquitous. 72 W, single-slot, and the default cost-efficient inference GPU across cloud providers (e.g. GCP G2). It represents how an embedding service would actually be deployed at scale, not a research-cluster fantasy.
  • Its constraints make optimization matter — which is the whole point. The L4's modest 300 GB/s bandwidth means the unfused-elementwise overhead found in Milestone 7 is a real fraction of runtime, so torch.compile fusion (1.63×) and the Triton pooling kernel have visible payoff. On an H100 you would be less memory-pressured and the same wins would be smaller and harder to demonstrate. Optimizing on the constrained chip is what makes the engineering legible.
Provenance

Sources & caveats

Where the numbers come from

GPU specs: NVIDIA product briefs / architecture whitepapers (L4, A100, Hopper H100), the CUDA C++ Programming Guide and Blackwell Tuning Guide for per-SM occupancy limits, and TechPowerUp / chipsandcheese / Wikipedia for the Blackwell estimates NVIDIA has not officially disclosed. R100/Rubin: NVIDIA's GTC 2025 / CES 2026 roadmap as reported by Tom's Hardware and The Next Platform (288 GB HBM4, ~22 TB/s, ~50 PFLOPS NVFP4 — preliminary). TPU v7 / Trainium3: the Google Cloud TPU7x docs (192 GB HBM3e, 7.37 TB/s, 4,614 TF FP8) and AWS / HPCwire (144 GB HBM3e, 4.9 TB/s, 2.52 PF FP8). ESM2: the Science paper, bioRxiv preprint, Hugging Face model cards, and facebookresearch/esm discussion #414 (linked above).

  • Tensor TFLOPS are dense. Marketing headlines are usually 2× (2:4 sparsity).
  • The L4 row is measured, not estimated — 58 SMs read directly off this project's card via cudaDeviceGetAttribute (CUDA cores = 58 × 128). Any GPU reports its own SM count this way.
  • SM / CUDA-core counts marked * (Blackwell ~144/148/160) are community estimates only because those cards aren't on hand to query and NVIDIA's datasheets omit per-die counts; the parts are dual-die packages presented as one GPU.
  • R100 (Rubin) ‡ is unreleased. Every R100 cell is an NVIDIA roadmap/announced figure (GTC 2025 / CES 2026, target H2 2026), subject to change and not measurable. SM count, compute capability, and per-GPU dense BF16/FP8 throughput have not been published, so R100 is omitted from the SM, BF16, and training-time views.
  • TPU v7 & Trainium3 † are non-NVIDIA accelerators, GA but cloud-only (rented, not bought). They have no SM / warp / CUDA-core model, so they appear on the physical axes only (memory, bandwidth, throughput). Figures are vendor-published per-chip peaks; per-chip TDP — and Trainium3's BF16 rate — aren't published. They're left out of the GPU-anchored training-time estimate, since its ~36% MFU assumption is calibrated to a CUDA GPU and doesn't transfer across architectures.
  • Per-SM limits are per compute capability, identical across all cards sharing an architecture — they are a property of the ISA, not the SKU.
  • Training-time estimates are compute-only at a fixed ~36% MFU; see the warning above.
