Milestone 6 · Triton masked-mean-pooling kernel

A hand-written Triton kernel fuses masked mean pooling into a single fp32-accumulating pass, validated against the Milestone 3 reference oracle and benchmarked against the PyTorch paths. It is the fastest implementation in all 6 shapes and numerically near-exact — but with the honest caveat that pooling is a tiny fraction of encoder time, so this does not move end-to-end throughput.

What was built

The kernel

  • src/esm2_perf/triton_pooling.py — triton_masked_mean_pool + the @triton.jit kernel. Grid (B, ⌈H/BLOCK_H⌉); each program owns one batch row and a block of hidden columns, streams over the sequence axis in BLOCK_T chunks, accumulating the masked sum and token count in fp32, then divides (count clamped to 1 so all-padding rows pool to zero, not NaN).
  • scripts/05_bench_triton_pooling.py — benchmarks pytorch (vectorized), einsum, compiled, and triton, recording latency, speedup, and max abs error vs the fp32 reference.
  • 8 GPU+Triton tests against the M3 per-sequence oracle (53 tests pass total; Triton tests skip on CPU-only machines).

What the benchmark says

Key takeaways

  • Fastest and most accurate at once. Triton wins every cell, and because it accumulates in fp32 its output is essentially the fp32 reference (err ~1e-8) while bf16 PyTorch pooling carries ~1e-3. No speed/accuracy trade-off here.
  • Read the 5.4× honestly. The eager vectorized masked_mean_pool materializes a full (B,T,H) temporary and is memory-bound — at seq 512 / batch 32 it balloons to 0.46 ms. Against the best PyTorch option (usually einsum, no temporary), Triton is a more modest, consistent 1.1×–1.6×.
  • einsum and compiled cross over. Slower than the simple vectorized path at small shapes (fixed overhead on a ~0.1 ms op), faster at the large memory-bound shapes. torch.compile does not help a tiny pointwise reduction — the same overhead story as Milestone 5.
  • Triton latency is nearly flat (~0.073–0.085 ms) until the largest cell (0.39 ms) — it is launch/overhead-bound below that.
Honest framing

Fast, correct — and end-to-end irrelevant

Pooling is ~0.08 ms of a ~25–225 ms forward

None of these speedups move pipeline throughput. The kernel earns its place as a correct, unit-tested Triton reduction validated against an independent oracle, and as a pooling path that is both faster and ~5 orders of magnitude more accurate than default bf16 pooling — useful if pooled embeddings are stored and compared downstream. It is not a claim that a custom kernel is the right tool for this op in a real pipeline. It is also not autotuned (fixed block sizes), so the largest-cell number may have room left.
