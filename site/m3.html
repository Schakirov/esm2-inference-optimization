Milestone 3 · Correctness & pooling harness

Implement masked mean pooling over real amino-acid tokens, then validate the fp16/bf16 paths against an fp32 eager reference with documented numerical tolerances. This establishes the correctness foundation that every later speed claim is checked against.

What this milestone will do

Plan

  • Masked mean pooling. Reduce per-token hidden states to one vector per protein by averaging over real amino-acid tokens only — excluding padding and the <cls>/<eos> special tokens.
  • fp32 eager reference. Establish a trusted reference output, then compare bf16 and fp16 embeddings against it with explicit, documented tolerances (exact equality is not expected across dtypes).
  • Implementation equivalence. Validate that alternative pooling implementations agree with the reference, so later optimized versions (incl. the Triton kernel) have a ground truth.
Planned outputs

Artifacts

  • src/esm2_perf/pooling.py — masked mean pooling implementation(s).
  • scripts/02_check_correctness.py — dtype-vs-reference comparison driver.
  • Unit tests asserting pooling correctness and tolerance bounds.
Why it matters

The point of doing this before more speed work

The baseline (Milestone 2) showed bf16 ≈ fp16 on throughput — but throughput parity says nothing about accuracy. Before claiming any optimization is "free," we need a documented numerical budget: how far can a pooled embedding drift from the fp32 reference and still be acceptable? Everything downstream — batching, torch.compile, the Triton kernel — is validated against the harness built here.
